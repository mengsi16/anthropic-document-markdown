# Demystifying Evals for AI Agents

**作者**: Mikaela Grace, Jeremy Hadfield, Rodrigo Olivares, Jiri De Jonghe 等  
**发布日期**: 2026年1月9日  
**来源**: [Anthropic Engineering](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)

---

## 概述

使代理有用的能力也使其难以评估。跨部署工作的策略结合多种技术来匹配它们测量的系统复杂性。

良好的评估帮助团队更有信心地发布AI代理。没有它们，很容易陷入反应性循环——仅在生产中捕捉问题，修复一个失败会创建另一个。评估在问题影响用户之前使问题和行为变化可见，其价值在代理的生命周期中复合增长。

## 评估的结构

### 基本定义

评估（"eval"）是AI系统的测试：给AI一个输入，然后将评分逻辑应用于其输出以测量成功。

#### 单轮 vs. 多轮评估

| 类型 | 特点 |
|------|------|
| **单轮评估** | 简单：提示、响应、评分逻辑。早期LLM的主要评估方法 |
| **多轮评估** | 越来越普遍。代理使用工具跨越多轮、修改状态并适应 |

#### Agent评估的复杂性

Agent评估更加复杂：
- 代理跨越许多轮使用工具
- 在环境中修改状态
- 适应中间结果
- 错误可能传播和复合
- **前沿模型可能找到超越静态评估限制的创造性解决方案**

*示例*: Opus 4.5 通过发现策略中的漏洞解决了关于预订航班的τ2-bench问题——它按书面标准"失败"了评估，但实际上为用户想出了更好的解决方案。

### 核心术语定义

| 术语 | 定义 |
|------|------|
| **Task (任务)** | 具有定义输入和成功标准的单个测试 |
| **Trial (试验)** | 对任务的每次尝试（因模型输出在运行间变化） |
| **Grader (评分器)** | 逻辑，对代理性能的某些方面进行评分。一个任务可以有多个评分器，每个包含多个断言（checks） |
| **Transcript (转录)** | 试验的完整记录，包括输出、工具调用、推理、中间结果等。对于Anthropic API，这是eval运行结束时的完整messages数组 |
| **Outcome (结果)** | 试验结束时环境的最终状态 |
| **Evaluation Harness (评估框架)** | 端到端运行eval的基础设施：提供指令和工具、并发运行任务、记录所有步骤、评分输出、聚合结果 |
| **Agent Harness/Scaffold (代理框架)** | 使模型能够充当代理的系统：处理输入、编排工具调用、返回结果。评估"代理"时，我们评估的是框架和模型的协作工作 |
| **Evaluation Suite (评估套件)** | 为测量特定能力或行为而设计的任务集合 |

## 为什么构建评估？

### 发展阶段

**早期阶段**: 团队通过手动测试、dogfooding和直觉的结合可以走得相当远。更严格的评估甚至看起来像减慢发布的开销。

**转折点**: 一旦代理投入生产并开始扩展，没有评估的构建开始崩溃：
- 用户报告代理感觉变更后变差了
- 团队在没有方法验证的情况下"盲目飞行"，只能猜测和检查
- 无法区分真实回归和噪声
- 不能在发布前自动测试数百种场景
- 无法测量改进

**实际案例**:
- **Claude Code**: 开始时基于反馈快速迭代；后来添加了evals（先是简明领域如简洁性和文件编辑，然后是复杂行为如过度工程）
- **Descript**: 为视频编辑代理围绕三个维度构建evals：不要破坏事情、做我要求的、做好它。从手动评分演变为LLM评分器
- **Bolt AI**: 已经有广泛使用的代理后3个月构建了eval系统，使用静态分析、浏览器agent测试、LLM评判

**关键洞察**:
- Evals在任何阶段都有用：
  - 早期：强制产品团队明确成功的含义
  - 后期：保持一致的质量门槛
- Evals还塑造你采用新模型的速度：
  - 没有evals的团队面临数周测试
  - 有evals的团队可以快速确定模型优势、调整提示、几天内升级

## 如何评估AI代理

### 常见代理类型及评估技术

目前部署的大规模代理类型：
- 编码代理
- 研究代理
- 计算机使用代理
- 对话代理

每种类型可使用相似技术评估。

### 评分器类型

Agent评估通常结合三种类型的评分器：

#### 1. Code-based Graders（基于代码的评分器）

| 方法 | 优势 | 劣势 |
|------|------|------|
| • 字符串匹配检查（精确、正则、模糊） | • 快速 | • 对有效变化脆弱 |
| • 二元测试（fail-to-pass, pass-to-pass） | • 便宜 | • 缺乏细微差别 |
| • 静态分析（lint, type, security） | • 客观 | • 有限用于某些主观任务 |
| • 结果验证 | • 可重现 | • |
| • 工具调用验证 | • 易调试 | • |
| • 转录分析 | • 验证特定条件 | • |

#### 2. Model-based Graders（基于模型的评分器）

| 方法 | 优势 | 劣势 |
|------|------|------|
| • 基于规则的评分 | • 灵活 | • 非确定性 |
| • 自然语言断言 | • 可扩展 | • 比代码昂贵 |
| • 成对比较 | • 捕捉细微差别 | • 需要与人类校准 |
| • 基于参考的评估 | • 处理开放任务 | • |
| • 多评委共识 | • 处理自由形式输出 | • |

#### 3. Human Graders（人类评分器）

| 方法 | 优势 | 劣势 |
|------|------|------|
| • SME审查 | • 金标准质量 | • 昂贵 |
| • 众包判断 | • 匹配专家用户判断 | • 慢 |
| • 抽样检查 | • 用于校准基于模型的评分器 | • 通常需要大规模访问人类专家 |
| • A/B测试 | • | • |
| • 评注者间一致性 | • | • |

**评分方式选择**:
- 加权（组合分数必须达到阈值）
- 二元（所有评分器必须通过）
- 混合

### 能力评估 vs. 回归评估

| 类型 | 问题 | 目标 |
|------|------|------|
| **Capability/Eval** | "这个代理能做什么好？" | 低通过率，针对代理挣扎的任务，给团队攀登的山峰 |
| **Regression Eval** | "代理还能处理它以前能处理的所有任务吗？" | 近100%通过率，防止倒退 |

**演进**: 代理启动并优化后，高通过率的能力eval可以"毕业"成为持续运行的回归套件。

## 各类型代理的具体评估方法

### 1. 编码代理（Coding Agents）

**特点**: 编写、测试和调试代码，导航代码库和运行命令

**自然选择**: 确定性评分器（软件通常直接评估：代码能否运行？测试是否通过？）

**广泛使用的基准**:
- **SWE-bench Verified**: 从流行Python仓库给代理GitHub issue，通过运行测试套件评分。解决方案只有在修复失败的测试且不破坏现有测试时才通过。
  - LLM一年内从40%进展到>80%
- **Terminal-Bench**: 测试端到端技术任务（如从源码构建Linux内核或训练ML模型）

**建议**: 
- 有pass-or-fail tests验证编码任务的关键结果后，通常有用的是也对transcript评分
- 启发式代码质量规则可以基于不仅仅是通过测试来评估生成的代码
- 基于模型的评分器和清晰规则可以评估行为如代理如何调用工具或与用户交互

**理论示例 - 认证绕过修复任务**:

```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token
```

### 2. 对话代理（Conversational Agents）

**特点**: 在支持、销售或教练等领域与用户交互，维护状态，中途使用工具和采取行动

**独特挑战**: 交互本身的质量是你评估的一部分

**有效评估通常依赖**:
- 可验证的最终状态结果
- 同时捕获任务完成和交互质量的规则
- **经常需要第二个LLM模拟用户**

**成功可能是多维度的**:
- 工单是否解决（状态检查）
- 是否在<10轮内完成（transcript约束）
- 语气是否适当（LLM规则）

**基准示例**: τ-Bench 和 τ2-Bench（模拟零售支持和航空预订的多轮交互）

**理论示例 - 客户支持退款处理**:

```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics:
      - n_turns
      - n_toolcalls
      - n_total_tokens
  - type: latency
    metrics:
      - time_to_first_token
      - output_tokens_per_sec
      - time_to_last_token
```

### 3. 研究代理（Research Agents）

**特点**: 收集、综合和分析信息，然后产生答案或报告等输出

**独特挑战**:
- 专家可能不同意综合是否全面
- 真相随着参考内容不断变化而转移
- 更长、更开放的输出有更多错误空间

**策略**: 结合评分器类型
- **基础性检查**: 验证声明得到检索来源支持
- **覆盖范围检查**: 定义良好答案必须包含的关键事实
- **源质量检查**: 确认咨询的源是权威性的，而不仅仅是第一个检索到的

**基准示例**: BrowseComp（测试AI代理能否在整个开放网络中找到大海捞针的问题）

**建议**: 
- 对于客观正确答案（"X公司Q3收入是多少？"），精确匹配有效
- LLM可以标记无根据的声明和覆盖范围的缺口
- 也可以验证开放性综合的连贯性和完整性
- **基于LLM的规则应频繁与专家人类判断校准**

### 4. 计算机使用代理（Computer Use Agents）

**特点**: 通过与人类相同的界面与软件交互——截图、鼠标点击、键盘输入、滚动——而非API或代码执行

**评估要求**:
- 在真实或沙盒环境中运行代理
- 检查是否达到预期结果

**基准示例**:
- **WebArena**: 测试基于浏览器的任务，使用URL和页面状态检查 + 后端状态验证
- **OSWorld**: 扩展到完整操作系统控制，完成后检查多样化产物（文件系统状态、应用配置、数据库内容、UI元素属性）

**浏览器使用平衡**:
| 方式 | 特点 |
|------|------|
| DOM-based interactions | 执行快但消耗很多token |
| Screenshot-based interactions | 较慢但更token高效 |

**实际案例 - Claude for Chrome**: 开发evals来检查代理是否为每个上下文选择了正确的工具 → 能够更快更准确地完成基于浏览器的任务

## 代理评估中的非确定性思维

**核心挑战**: 无论代理类型如何，代理行为在运行间变化，使评估结果比初看更难解释

### 两个关键指标

#### pass@k
- **衡量**: 代理在k次尝试中至少获得一次正确解的可能性
- k增加时，pass@k上升：更多"射门机会"意味着至少1次成功的几率更高
- **50% pass@1** = 模型在一半的任务上首次尝试成功
- **适用场景**: 编码中通常最关心第一次尝试找到解决方案

#### pass^k
- **衡量**: 所有k次试验都成功的概率
- k增加时，pass^k下降：要求更多试验的一致性是更高的门槛
- **示例**: 如果代理75%的单试验成功率，运行3次试验，所有三次通过的概率 = (0.75)³ ≈ 42%
- **适用场景**: 面向客户的代理，用户期望每次都有可靠的行为

**差异**: k=1时两者相同（等于单试验成功率）。k=10时讲述相反的故事：pass@k接近100%而pass^k降到0。

**选择指导**:
- **pass@k**: 一个成功就重要的工具
- **pass^k**: 一致性至关重要的代理

## 从零到一：伟大评估路线图

### 第一阶段：收集初始评估数据集任务

#### Step 0: 尽早开始
❌ 不要因为认为需要数百个任务而推迟  
✅ **20-50个简单任务**（来自真实失败）是很棒的开始  
**理由**: 早期代理开发中每个系统变更通常有清晰明显的影响，大效应量意味着小样本量就足够

**警告**: Evals越等越难建。早期产品需求自然转化为测试案例。太晚则要从实时系统反向工程成功标准

#### Step 1: 从已有手动测试开始
- 开发期间运行的手动检查
- 发布前验证的行为
- 最终用户的常见任务
- 如果已生产：查看bug追踪和支持队列
- 将用户报告的故障转换为测试案例确保套件反映实际使用

#### Step 2: 编写明确任务和参考解决方案
**好任务的标志**: 两个领域专家独立达到相同的pass/fail裁决  
**检查**: 他们自己能否通过？如果不能，任务需要改进

⚠️ **模糊性 = 指标噪声**

**要求**:
- 每个任务应该可以被正确遵循指令的代理通过
  - *微妙*: 如果任务要求写脚本但不指定filepath，测试假设特定filepath，代理可能非自身原因失败
- 评分器检查的所有内容应该从任务描述清楚
- 创建**参考解决方案**：已知通过的输出证明任务可解并验证评分器配置正确
- **前沿模型0% pass@100** = 最常是坏任务信号而非无能代理信号

#### Step 3: 构建平衡问题集
- 测试行为应该发生和不应该发生的案例
- 单方面eval导致单方面优化
- 尝试避免类别不平衡evals
- **实际教训** (Claude.ai web search evals):
  - 应该搜索的查询（如找天气）
  - 不应该搜索的查询（如"谁创立了Apple？"）
  - 平衡欠触发和过触发需要多轮提示和eval改进

### 第二阶段：设计评估框架和评分器

#### Step 4: 构建健壮的评估框架和稳定环境
**关键**:
- 评估中的代理功能大致与生产中使用的代理相同
- 环境本身不引入进一步噪声
- **每次试验应该"隔离"**：从干净环境开始
- 运行间不必要的共享状态可能导致因基础设施不稳定而非代理性能的相关故障
- 共享状态也可能人为膨胀性能（如Claude通过检查先前试验的git历史获得不公平优势）

#### Step 5: 思考设计评分器
**推荐优先级**: 
1. 可能的地方选择确定性评分器
2. 必要或需要额外灵活性的地方使用LLM评分器
3. 审慎地使用人类评分器进行额外验证

**常见陷阱和建议**:

❌ **避免**: 检查非常特定的步骤序列（如正确的工具调用顺序）→ 太僵化，过度脆弱  
✅ **推荐**: 评估代理生产的内容，而非它走的路径

✅ **部分信用**: 对于多组件任务，构建部分信用机制

✅ **仔细迭代验证准确度**: LLM-as-judge应与人类专家密切校准

✅ **给LLM退路**: 提供指令在信息不足时返回"Unknown"

✅ **创建清晰的、结构化的规则**: 分离维度，每个维度使用隔离的LLM评判

⚠️ **警惕微妙的失败模式**: 即使复杂的团队也可能错过这些问题
- **示例1**: Opus 4.5最初CORE-Bench得分42%，直到发现问题：僵化评分惩罚"96.12"期望"96.124991..."、模糊任务规范、不可能完全复现的随机任务。修复后跳至95%
- **示例2**: METR发现时间horizon基准中的多个误配任务：要求代理优化到声明的分数阈值，但评分要求超过阈值

✅ **双重检查**: 仔细双重检查任务和评分器可以帮助避免这些问题

✅ **防作弊**: 任务和评分器应设计为通过真正解决问题而非利用意外漏洞才能通过

### 第三阶段：长期维护和使用

#### Step 6: 检查转录
**除非你阅读许多试验的转录和评分，否则不会知道你的评分器工作得如何**

**投资**: Anthropic投资了查看eval transcripts的工具并定期花时间阅读它们
**判断标准**: 失败应该感觉公平：清楚代理哪里错了以及为什么
**价值**: 当分数不攀升时，我们需要信心是由于代理表现而非eval本身

#### Step 7: 监控能力eval饱和
**100%的eval跟踪回归但不提供改进信号**

**Eval饱和发生**: 当代理通过所有可解决任务时
**后果**: 进展将放缓，因为只有最困难的任务保留
**欺骗性**: 大能力改进表现为小分数增长

**示例**: Qodo代码审查初创公司最初对Opus 4.5印象不深刻，因为他们的一次性编码evals未捕获更长复杂任务的增益。回应：开发新的agentic eval框架，提供更清晰的进展图

**规则**: 直到有人深入eval细节并阅读一些transcripts才采信eval分数。如果评分不公平、任务歧义、有效解决方案受惩罚或框架限制模型，eval应该修改

#### Step 8: 通过开放贡献和维护长期保持评估套件健康
**评估套件是需要持续关注和明确所有权的活工件**

**经验**: 
- 最有效的方法：建立专门的evals团队拥有核心基础设施，同时领域专家和产品团队贡献大多数eval任务并自己运行评估
- AI产品团队拥有和迭代evals应像维护单元测试一样常规
- 团队可能在"早期测试工作正常"的AI特性上浪费数周但未能满足良好eval会早期暴露的未陈述期望
- 定义eval任务是压力测试产品需求是否足够具体以开始构建的最佳方法之一

**推荐实践**: **Eval驱动开发**
- 在代理能够满足之前构建evals以定义计划能力
- 迭代直到代理表现良好
- 内部经常构建今天"足够好"的特性，但对几个月后模型能做什么下注
- 以低通过率开始的capability evals使这可见
- 新模型发布时，运行套件快速揭示哪些赌注获胜

**赋能**: 最接近产品要求和用户的人最定位定义成功。让产品经理、客户成功经理或销售人员使用Claude Code贡献eval任务作为PR——让他们参与！或者更好，主动启用他们！

## 评估与其他方法的整合

自动化评估可以在数千任务中运行代理而无需部署生产或影响真实用户。但这只是理解代理性能的众多方式之一。

**完整图片包括**:
- 生产监控
- 用户反馈
- A/B测试
- 手动转录审查
- 系统化的人类评估

### 方法对比矩阵

| 方法 | 优点 | 缺点 |
|------|------|------|
| **自动化Evals** | 快速迭代；完全可重现；无用户影响；可在每次提交运行；无需prod部署即可大规模测试场景 | 需要前期投资构建；需持续维护以避免漂移；如果不匹配实际使用模式可能产生虚假自信 |
| **生产监控** | 揭示大规模真实用户行为；捕获合成evals错过的问题；提供代理实际表现的ground truth | 反应性；问题到达用户前不知道；信号可能有噪声；需要仪表盘投资；缺乏评分ground truth |
| **A/B测试** | 衡量真实用户结果（保留、任务完成）；控制混杂因素；可扩展系统化 | 慢；需要数天/周达到显著性且需要足够流量；只测试部署的更改；对指标变化的底层原因信号较少 |
| **用户反馈** | 显式信号（ thumbs-down、bug报告）；揭示没预料到的问题；带来真实人类用户的真实例子；反馈通常与产品目标相关 | 稀疏和自选择；偏向严重问题；用户很少解释为什么失败；非自动化；主要依赖用户捕捉问题可能有负面用户影响 |
| **手动转录审查** | 人类阅读代理对话；建立失败模式的直觉；捕捉自动检查错过的细微质量问题；帮助校准"好"的样子并掌握细节 | 费时；不可扩展；覆盖不一致；审阅者疲劳或不同审阅者可能影响信号质量；通常只提供定性信号而非清晰的定量评分 |
| **系统化人类研究** | 来自多个训练有素的评分者的金标准质量判断；处理主观或模糊任务；提供改进基于模型的评分器的信号 | 相对昂贵和周转慢；难频繁运行；评注者间分歧需要调和；复杂领域（法律、金融、医疗保健）需要专家进行研究 |

### 整合策略（瑞士奶酪模型）

像安全工程中的瑞士奶酪模型一样，**没有一个单一评估层能捕捉每个问题**。多层结合时，一层漏过的问题被另一层捕获。

**最有效的团队结合这些方法**:
- **自动化evals**: 快速迭代（预发布和CI/CD，作为第一道防线）
- **生产监控**: ground truth（发布后检测分布漂移和意外的真实世界失败）
- **A/B测试**: 验证重大更改（当有足够流量时）
- **用户反馈和转录审查**: 持续实践填补空白
- **系统化人类研究**: 校准LLM评分器或评估主观输出（人类共识作为参考标准时保留）

## 结论

没有evals的团队陷入反应性循环——修复一个失败、创建另一个、无法区分真实回归和噪声。早期投资的团队相反：开发加速因为失败变成测试案例、测试案例防止回归、指标取代猜测。

**模式因代理类型而异，但这里描述的基本原则保持不变**:

✅ 尽早开始，不要等待完美套件  
✅ 从看到的失败中获取现实任务  
✅ 定义明确、稳健的成功标准  
✅ 思考设计评分器并结合多种类型  
✅ 确保问题对模型足够困难  
✅ 迭代评估以提高信噪比  
✅ **阅读转录！**

**未来展望**: AI代理评估仍是一个新兴、快速发展的领域。随着代理承担更长任务、在多代理系统中协作、处理越来越主观的工作，我们将需要适应我们的技术。我们将继续分享更多最佳实践。

## 附录：评估框架

几个开源和商业框架可以帮助团队实现代理评估而不从头构建基础设施：

| 框架 | 特点 |
|------|------|
| **Harbor** | 设计用于在容器化环境中运行代理，具有跨云提供商大规模运行试验的基础设施以及定义任务和评分器的标准化格式。Terminal-Bench 2.0等流行基准通过Harbor registry分发 |
| **Braintrust** | 结合离线评估和生产可观察性及实验跟踪的平台——适合需要同时在开发中迭代和生产中监控质量的团队。`autoevals`库包含事实性、相关性等常见维度的预构建scorer |
| **LangSmith** | 提供跟踪、离线和在线评估以及数据集管理，与LangChain生态系统紧密集成 |
| **Langfuse** | 类似能力，作为自托管开源替代方案，适合有数据驻留要求的团队 |
| **Arize Phoenix** | 开源平台，用于LLM跟踪、调试和离线/在线评估；AX是扩展Phoenix用于规模、优化和监控的SaaS产品 |

**建议**: 许多团队结合多个工具、滚动自己的eval框架，或仅使用简单的评估脚本作为起点。虽然框架可以是加速进步和标准化的有价值方式，但它们只有在你运行的eval任务上才好用。**快速选择适合工作流的框架，然后通过迭代高质量测试案例和评分器将精力投入evals本身通常是最佳路径**。
