# Building Effective AI Agents

**作者**: Erik Schluntz 和 Barry Zhang  
**发布日期**: 2024年12月19日  
**来源**: [Anthropic Engineering](https://www.anthropic.com/engineering/building-effective-agents)

## 概述

我们与数十个团队合作构建跨行业的大语言模型（LLM）代理。最成功的实现并非使用复杂的框架或专用库，而是使用简单、可组合的模式构建。

本文分享了我们在与客户合作和构建代理过程中的经验教训，并为开发者提供了构建有效代理的实用建议。

## 什么是代理？

"代理"可以有多种定义方式：
- **工作流（Workflows）**: 通过预定义代码路径编排LLM和工具的系统
- **代理（Agents）**: LLM动态指导自身流程和工具使用的系统，保持对如何完成任务的自主控制

## 何时（以及何时不）使用代理

### 核心原则

**找到最简单的解决方案，仅在需要时增加复杂性**

- Agentic系统通常以延迟和成本换取更好的任务性能
- 对于许多应用，优化单次LLM调用（结合检索和上下文示例）通常足够

### 何时使用框架

可用的框架包括：
- **Claude Agent SDK**
- **AWS Strands Agents SDK**
- **Rivet** (拖放式GUI LLM工作流构建器)
- **Vellum** (GUI工具)

**建议**: 开发者应直接使用LLM API开始；如果使用框架，要理解底层代码。

## 构建模块、工作流和代理

### 构建基础：增强型LLM

基本构件是通过检索、工具和内存等功能增强的LLM。

**关键方面**:
1. 针对特定用例定制能力
2. 确保为LLM提供易于使用且文档完善的接口

推荐通过 **Model Context Protocol (MCP)** 实现。

### 工作流模式

#### 1. 提示链式（Prompt Chaining）

将任务分解为步骤序列，每个LLM调用处理前一个的输出。

```
输入 → LLM → 检查门 → LLM → 输出
```

**适用场景**: 任务可以轻松清晰地分解为固定子任务时。

**示例**:
- 生成营销文案，然后翻译成不同语言
- 编写文档大纲，检查标准，然后基于大纲编写完整文档

#### 2. 路由（Routing）

分类输入并引导至专门的后续任务。

```
输入 → 分类器 → 任务A/任务B/任务C
```

**适用场景**: 复杂任务有不同类别需要分别处理时。

**示例**:
- 将不同类型的客户服务查询路由到不同流程
- 将简单问题导向Haiku模型，复杂问题导向Sonnet模型

#### 3. 并行化（Parallelization）

两种关键变体：
- **分段（Sectioning）**: 将任务分解为并行运行的独立子任务
- **投票（Voting）**: 多次运行相同任务获得多样化输出

**适用场景**: 子任务可以并行化以提高速度，或需要多种视角提高结果置信度时。

**示例**:
- 实施护栏：一个模型处理查询，另一个检查不当内容
- 自动化评估：每个LLM调用评估性能的不同方面

#### 4. 编排者-工作者（Orchestrator-Workers）

中央LLM动态分解任务，委托给工作者LLM，综合结果。

**适用场景**: 复杂任务中无法预测所需子任务时。

**示例**:
- 需要对多个文件进行复杂更改的编码产品
- 涉及从多个来源收集和分析信息的搜索任务

#### 5. 评估者-优化者（Evaluator-Optimizer）

一个LLM生成响应，另一个提供评估反馈的循环。

**适用场景**: 有明确评估标准，迭代改进能带来可衡量价值时。

**示例**:
- 文学翻译（可能有细微差别需要捕捉）
- 需要多轮搜索和分析的复杂搜索任务

### 自主代理

代理在成熟的关键能力下出现在生产环境中：
- 理解复杂输入
- 推理和规划
- 可靠地使用工具
- 从错误中恢复

**代理工作流程**:
1. 从人类用户接收命令或进行交互讨论
2. 规划并独立运行
3. 在检查点或遇到障碍时可暂停等待人工反馈
4. 通常在完成任务时终止，或包含停止条件

**适用场景**: 开放性问题，难以或无法预测所需步骤数量时。

**示例**:
- 解决SWE-bench任务的编码Agent（涉及多文件编辑）
- "计算机使用"参考实现（Claude使用计算机完成任务）

## 组合和定制模式

这些构建模块不是规定性的，而是开发者可以根据用例塑造和组合的常见模式。

**成功的关键**: 测量性能并迭代实现。只有当简单解决方案明显不足时才添加复杂性。

## 总结：三个核心原则

1. **保持简洁性** - 维护代理设计的简洁性
2. **优先透明度** - 明确显示代理的规划步骤
3. **精心设计ACI** - 通过彻底的工具文档和测试精心设计代理-计算机接口（ACI）

## 附录1：实践中的代理应用

### A. 客户支持

客户支持自然适合开放性代理：
- 支持交互自然遵循对话流程
- 可集成工具拉取客户数据、订单历史等
- 操作（退款、更新工单）可通过编程处理
- 成功可通过用户定义的解决率清晰测量

### B. 编码代理

软件开发领域显示出巨大潜力：
- 代码解决方案可通过自动化测试验证
- 代理可使用测试结果作为反馈进行迭代
- 问题空间明确定义且结构化
- 输出质量可客观测量

**实际成果**: 代理现在可以在SWE-bench Verified基准上仅基于PR描述解决真实的GitHub问题。

## 附录2：提示工程你的工具

### 工具格式设计建议

1. **给模型足够的token来"思考"**，避免陷入困境
2. **保持格式接近模型在互联网文本中自然看到的格式**
3. **确保没有格式化开销**（如必须保持数千行代码的准确计数）

### ACI设计最佳实践

**思考规则**: 像对待人机接口（HCI）一样投入精力创建良好的代理-计算机接口（ACI）：

1. **换位思考** - 从模型角度看工具是否明显可用？
   - 良好的工具定义应包含：示例用法、边缘情况、输入格式要求、与其他工具的清晰边界

2. **优化参数名称和描述** - 使其显而易见
   - 就像为初级开发人员编写优秀的docstring
   - 使用多个相似工具时尤为重要

3. **测试工具使用情况** - 运行许多示例输入查看错误模式
   - 使用workbench测试
   - 迭代改进

4. **防错设计（Poka-yoke）** - 更改参数使犯错更难
   - 例如：要求绝对文件路径而非相对路径

**经验分享**: 在构建SWE-bench代理时，实际上花在优化工具上的时间比整体提示更多。例如发现模型在使用相对文件路径时会出错后，改为始终要求绝对路径——模型随后完美使用了此方法。
